# ARCHITECTURE.md

## 一、整体架构定位

Financial Research Assistant 的整体架构不是普通的“文件上传 + AI 总结”结构，而是围绕研究过程构建的分层系统。

其核心逻辑是：

- 原始材料进入系统
- 解析为统一文本与页面结构
- 提取结构化信息
- 生成研究报告
- 结合历史报告与最新材料更新研究结论
- 对未来做预测
- 用真实数据做回测
- 把偏差写回系统记忆
- 最终给出管理层决策支持

系统本质是一条 **研究状态链**。

---

## 二、系统主流程

### 1. 上传层（upload）
负责上传原始材料。

材料可能包括：
- 年报
- 中报 / 半年报
- 季报
- 业绩公告
- 业绩演示材料
- 电话会纪要
- 并购公告
- 投资者沟通材料

---

### 2. 解析层（parse）
目标：从 PDF 中获取可用的文本与页面信息。

当前解析逻辑大致分为：

#### 快速主通道
- 以 PyMuPDF 为主
- 快速抽取全文与页面文本

#### 文本补救通道
- 对疑似差页再尝试：
  - pdfplumber
  - pypdf

#### 复杂页处理通道
- 对扫描页、图表混排页、图片页等，进入：
  - 图片渲染
  - 多模态 / 视觉模型解析

#### 输出
- `parsed_*.json`

parsed 结果应至少包含：
- full_text
- pages
- 元数据
- 页面级解析摘要
- 当前解析策略摘要

---

### 3. 提取层（extract）
目标：从 parsed 结果中提取结构化信息。

提取内容包括但不限于：
- period_key
- report_type
- document_type
- 关键片段
- 关键信息摘要
- 材料时间戳
- 是否主财报
- 是否可调整预测

#### 输出
- `extracted_*.json`

extract 层原则：
- 尽量结构化
- 尽量保留来源
- 不要在此层做过重主观分析

---

### 4. 指标层（metrics / actuals）
目标：从解析和提取结果中抽取财务指标，并形成标准化口径。

#### metrics
负责：
- 自动抽取财务指标
- 单位换算
- 同比 / 环比增强
- 行业画像辅助下的重点指标识别

#### actuals
负责：
- 生成 actual metrics registry
- 只允许主时序正式财报进入 actuals
- 辅助材料不得进入主时序最终实际值

这一层非常关键，因为它决定：
- forecast_check 的对照基础
- 未来回测是否可信

---

### 5. 分析层（analyze）
目标：基于 parsed + extracted + 多材料上下文，生成一份结构化财报研究报告。

analyze 的输出不是简单文本，而应优先是 JSON 结构，再转换为展示层 markdown。

#### 输入
- parsed JSON
- extracted JSON
- 公司画像
- 多材料上下文
- 行业标签
- 主时序锚点材料

#### 输出
- `report_*.json`
- Markdown 展示结果

analyze 的核心意义：
- 站在某个时点，根据若干材料，生成“当时视角”的研究报告

---

### 6. 更新层（update）
目标：生成最新版研究报告。

#### update 的输入
- 历史全部材料
- 历史全部报告
- history memory
- 上一次报告
- 最新一期材料（主财报或辅助材料）

#### update 的输出
- `master_report`
- `delta_report`
- 本期相对上期预期变化摘要

#### update 的核心定义
update 不是“改旧报告”，而是：

> 站在某个时间点，把当时全部已知信息重新汇总，形成一份新研究报告。

这一步必须纳入历史信息，否则累计误差会被忽略。

---

### 7. 预测层（forecast）
目标：基于当前信息状态，对下一期/下一年进行预测。

#### 当前原则
- 统计模型为主
- AI 解释为辅
- 需要 snapshot
- 需要 forecast registry
- 需要情景分析
- 需要可回测

#### 预测与 update 的区别
- update：描述“截至当前”的最优研究状态
- forecast：判断“从当前走向未来”的路径是否仍然成立

---

### 8. 回测层（forecast_check / backtest）
目标：用真实值对照历史预测。

#### 需要回答的问题
- 预测对不对
- 偏差有多大
- 偏差来自哪里
- 哪些信号有效
- 哪些信号失效
- 框架是否需要修正

#### 输出
- `forecast_check_*.json`
- 回测摘要
- 偏差趋势
- revision log
- history memory 更新

---

### 9. 问答检索层（qa）
目标：基于历史 parsed / extracted / report / forecast_check 进行检索与问答。

要求：
- 有引用来源
- 资料不足时明确说明
- 不能编造

---

### 10. Repository / Timeline 层
目标：
- 展示材料总览
- 展示时间轴
- 展示 SQLite 轻量索引
- 展示 parsed / extracted / report 状态

这层既是资料仓库，也是调试总览页。

---

## 三、正式版与测试版架构

### 正式版
面向日常操作。

原则：
- 操作尽量简化
- 后台逻辑不简化
- 减少中间细节暴露

当前正式版主要页面：
- 工作台
- 公司画像

### 测试版
面向调试和开发。

特点：
- 每一个处理层都是独立页面
- 便于查看中间 JSON
- 便于 debug 与单层回归测试

---

## 四、模型接入架构

当前项目不希望业务层直接依赖某一个模型厂商。

### 当前设计思路
- `agent_registry.json`：定义业务角色到 profile 的映射
- `llm_profiles.json`：定义模型档案
- `llm_gateway_service.py`：统一模型调用网关

### 设计目的
以后模型可能会从：
- 阿里云
- DeepSeek
- 其他更强/更便宜模型

之间频繁切换，因此必须做到：
- Agent 只认 profile
- 业务层不直接写死 provider
- 模型切换尽量不动业务代码

---

## 五、公司画像架构

公司画像是整个研究系统的重要上层结构，不是普通标签装饰。

### 当前维度设计
- BM（商业模式）— 单选
- VC（价值链位势）— 单选
- LC（生命周期）— 单选
- LC.sub_type（生命周期修饰语）— 可选
- G（特殊因子）— 多选

### 当前原则
- 手动覆盖优先级最高
- analyze / update 时自动刷新 auto profile snapshot
- 自动结果与手动结果冲突时，只提示，不自动覆盖

---

## 六、任务运行架构

当前正式版工作台已经开始从同步执行，向后台任务模式过渡。

### 现状
已具备：
- 任务状态文件
- 进度展示
- 协作式取消
- 后台线程执行

但尚未具备：
- 强制中断
- 完整任务队列
- 断点恢复
- 真正统一的 workflow registry

这一块后续仍需重构和加强。

---

## 七、当前架构中最不能被破坏的约束

1. 主财报与辅助材料必须分开。
2. 主时间轴不能被辅助材料打乱。
3. actuals 只能来自主财报。
4. update 必须纳入历史全部信息，不能只看最新一期。
5. 公司画像手动覆盖优先级最高。
6. 任何 JSON schema 变更都必须考虑历史兼容。
